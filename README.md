## About Me

I spent over a decade working on the hospital floor in Bengaluru. Over those years, I managed daily operations, coordinated between clinical departments, handled billing, and worked with patient data entry systems. 

I learned firsthand that when clinical software fails, it isn't because of a bug in the code. It is because the designers did not understand how a hospital actually runs. The admission clerk who has ten patients waiting in line will make typos. The doctor who needs to search a history needs quick answers without security alerts.

Now, I work as a Business Analyst and Product Owner to help engineering teams build software that fits clinical realities. I write PRDs, define user stories with clear acceptance criteria, and map out database structures that match healthcare standards.

---

## Core Focus Areas

### 1. Healthcare Data Management & Standardization
Mapping different data formats — like CSV files from OPD desks or PDFs from radiology — into consistent database tables. Making sure clinical data matches standard formats like FHIR and ABDM requirements.

### 2. Operational Data Integrity (ABDM 2.0 Workflows)
Designing requirements and data flows that match national digital health standards, including Ayushman Bharat Digital Mission (ABDM) patient identity verification and ABHA number mapping.

### 3. Data Privacy & DPDP Compliance
Writing specifications that enforce patient consent rules and data security at every stage. I design logic to handle PII masking, data quarantine, and automatic record deletions when a patient withdraws consent under the DPDP Act 2023.

## Featured Projects

### OmniIngest Clinical Data Engine (ABDM 2.0)
A clinical middleware designed to catch data entry errors (like missing ABHA IDs or incorrect spelling) before they reach the main database.
* **Repository:** [nisar46/OmniIngest-Clinical-Engine](https://github.com/nisar46/OmniIngest-Clinical-Engine)
* **Functional Scope:** Standardizes clinical data, manages consent records, and uses a hybrid schema design to prevent system crashes when different departments use different fields.
* **Key Artifacts:** PRD v1.0, 12 Jira user stories, data flow diagram, and UAT checklist.

### Clinosyn Healthcare OS
An offline search terminal that allows doctors to search patient records using natural language queries without sending any patient data to cloud services.
* **Repository:** [nisar46/Clinosyn-Healthcare-OS](https://github.com/nisar46/Clinosyn-Healthcare-OS)
* **Functional Scope:** Translates simple search queries into SQL commands, masks patient names on screen by default, and includes a "Kill-Switch" feature to completely purge records when a patient withdraws consent.
* **Key Artifacts:** PRD v1.0, 8 Jira user stories, NL-to-SQL logic map, and Confluence documentation.
